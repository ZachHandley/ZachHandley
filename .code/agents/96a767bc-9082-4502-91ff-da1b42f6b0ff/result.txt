### **Performance Optimization & Refinement Plan**

This plan is structured in prioritized phases, starting with low-risk, high-impact changes and progressing to more advanced optimizations. All suggestions are designed to be implemented behind feature flags for safe A/B testing.

---

### **1. Prioritized Optimization Plan**

#### **Phase 1: Quick Wins (Low-Risk, High-Impact)**

These changes focus on reducing the initial bundle size and GPU load with minimal visual or logical modifications.

1.  **Code Splitting & Lazy Loading:** The ~1.74MB `LinkApp` chunk is the biggest bottleneck. We can split it by dynamically importing heavy components.
    *   **Action:** Use Svelte's `{#await}` block or Threlte's `<Suspense>` to lazy-load components that are not immediately visible or essential. The main scene itself can be lazy-loaded.
    *   **Implementation (`index.astro`):**
        ```astro
        ---
        // Lazy load the main 3D component
        import { lazy } from 'svelte';
        const LinkApp = lazy(() => import('../components/svelte/LinkApp.svelte'));
        ---
        <Fragment set:html={
          LinkApp.render({ links: serverLinks })
        } />
        ```
    *   **Implementation (within `LinkApp.svelte`):** Lazy load secondary models or UI elements.
        ```svelte
        <script>
          import { Suspense } from '@threlte/core';
          import LoadingScreen from './ui/LoadingScreen.svelte';

          const Castle = lazy(() => import('./scene/Castle.svelte'));
        </script>

        <Suspense fallback={LoadingScreen}>
          <Castle />
        </Suspense>
        ```

2.  **Texture Compression (KTX2/Basis):** PNGs and JPEGs are not GPU-native formats. KTX2 with Basis Universal compression offers smaller file sizes and dramatically lower VRAM usage.
    *   **Action:** Convert all `.png`, `.webp`, and `.jpg` textures to `.ktx2`. Use a tool like `gltf-transform` or `basis_universal`.
    *   **Implementation:** Update your asset pipeline and loader.
        ```typescript
        // In assetManager.svelte.ts
        import { KTX2Loader } from 'three/examples/jsm/loaders/KTX2Loader.js';

        const ktx2Loader = new KTX2Loader()
          .setTranscoderPath('/basis_transcoder/') // Host the transcoder files in /public
          .detectSupport(renderer);

        // When loading textures, use ktx2Loader
        ```
    *   **Flag:** `?texture_format=ktx2` vs `?texture_format=png`.

3.  **Renderer & Canvas Settings:** Simple tweaks can yield significant performance gains, especially on high-DPI screens.
    *   **Action:** In `BaseScene.svelte`, clamp the device pixel ratio (DPR) and set a high-performance power preference.
    *   **Implementation (`BaseScene.svelte`):**
        ```svelte
        <Canvas
          powerPreference="high-performance"
          dpr={[1, 1.5]} <!-- Clamp DPR to a max of 1.5 -->
        >
          <!-- ... -->
        </Canvas>
        ```
    *   **Fallback:** Lower-end devices will benefit most. A DPR of `1` is a safe fallback.

#### **Phase 2: Medium-Risk (Toggleable Enhancements)**

These changes involve more significant logic but can be toggled for comparison.

1.  **Instancing for Crates:** Instead of rendering hundreds of individual `Mesh` objects for crates, use a single `InstancedMesh` to reduce draw calls from N to 1.
    *   **Action:** Create a new component, e.g., `InstancedCrates.svelte`, that takes positions from `crateController` and renders a Threlte `<InstancedMesh>`.
    *   **Implementation (`InstancedCrates.svelte`):**
        ```svelte
        <script lang="ts">
          import { T } from '@threlte/core';
          import { InstancedMesh } from '@threlte/extras';
          // Assume crate geometry and material are loaded once and passed as props
          export let geometry;
          export let material;
          export let positions: { x: number; y: number; z: number }[];
        </script>

        <InstancedMesh {geometry} {material} args={[geometry, material, positions.length]}>
          {#each positions as pos, i}
            <T.Object3D position={[pos.x, pos.y, pos.z]} instanceId={i} />
          {/each}
        </InstancedMesh>
        ```
    *   **Flag:** Conditionally render `<StackedLinks>` (old) vs. `<InstancedCrates>` (new).

2.  **Shadow Optimization:** Shadows are notoriously expensive.
    *   **Action:**
        1.  Reduce the shadow map size: `<DirectionalLight shadow.mapSize={[1024, 1024]} ... />`.
        2.  Tighten the shadow camera frustum to only cover the necessary area.
        3.  Use `@threlte/extras` `<ContactShadows>` for a cheap, high-quality ground shadow effect, and disable expensive `castShadow` on most objects.
    *   **Flag:** `?shadows=contact` vs. `?shadows=realtime`.

3.  **Mesh Optimization (Meshopt):** Meshopt compression is a perfect companion to Draco, further reducing buffer sizes.
    *   **Action:** Process your GLB/GLTF files with `gltf-transform optimize --compress=meshopt`. Then, provide the decoder to your `GLTFLoader`.
    *   **Implementation (`assetManager.svelte.ts`):**
        ```typescript
        import { MeshoptDecoder } from 'three/examples/jsm/libs/meshopt_decoder.module.js';

        const gltfLoader = new GLTFLoader();
        gltfLoader.setMeshoptDecoder(MeshoptDecoder);
        ```

#### **Phase 3: Advanced Optimizations (Optional)**

1.  **Level of Detail (LOD):** For complex models like the dragon or castle, serve lower-polygon versions when they are far from the camera.
    *   **Action:** Create 2-3 versions of your key models (e.g., `Dragon_LOD0`, `Dragon_LOD1`). Use Three.js's `LOD` class within a custom Threlte component.
    *   **Implementation (`DragonLOD.svelte`):**
        ```svelte
        <script>
          import { T, useThrelte } from '@threlte/core';
          import { LOD } from 'three/src/objects/LOD.js';
          // Assume models are loaded
          export let highDetailModel, mediumDetailModel;

          const { scene } = useThrelte();
          const lod = new LOD();
          lod.addLevel(highDetailModel, 20); // distance: 20
          lod.addLevel(mediumDetailModel, 80); // distance: 80
          scene.add(lod);
        </script>
        ```

2.  **Dynamic Resolution:** For scenes with heavy GPU load (e.g., many particles), dynamically lower the resolution to maintain FPS.
    *   **Action:** Create a `useTask` that monitors FPS. If it drops below a threshold (e.g., 45 FPS), reduce the renderer's pixel ratio.

### **2. Threlte-Focused Best Practices**

*   **`useTask`:** Continue using `useTask` for anything that runs every frame (camera updates, animations). Ensure you use the `auto-start/stop` and `pausable` options to save resources when the tab is inactive.
*   **Lazy Components & `<Suspense>`:** As mentioned in Phase 1, this is your primary tool for code splitting. Wrap any non-critical, heavy component in `<Suspense>`.
*   **`<T>` Component:** Use the `<T>` component to declaratively instantiate and manage Three.js objects. It helps with memoization and avoids unnecessary object recreation on component updates.
*   **Preloading:** Use `@threlte/extras` `usePreload` to load assets for the next "scene" or interaction in the background without blocking the main thread.
    ```svelte
    <script>
      import { usePreload } from '@threlte/extras';
      // Preload all assets needed for the modal that opens on crate click
      usePreload(['/models/CrateExplode-transformed.glb']);
    </script>
    ```

### **3. Three.js Specifics Checklist**

*   **Renderer:**
    *   `powerPreference: 'high-performance'`
    *   `dpr={[1, 1.5]}` (or even `[0.5, 1]` on low-end)
    *   Disable `antialias` on the `<Canvas>` if you use post-processing effects like TAA or SMAA.
*   **Geometries & Materials:**
    *   **Share:** Create geometries and materials *once* and reuse them. Pass them as props instead of re-declaring them in child components.
    *   **Dispose:** In Svelte's `onDestroy` lifecycle hook (or Threlte's `useTask` cleanup), call `.dispose()` on geometries and materials that are no longer needed to prevent memory leaks.
*   **Frustum Culling:** Enabled by default. Ensure your crate placement logic doesn't create one massive bounding box, which would defeat culling. Instancing helps here.

### **4. Crate & Link Placement Improvements**

The current grid layout is functional but can be improved for aesthetics and performance.

1.  **Avoid Occlusion:** The central dragon is a key visual. The layout should respect its space.
    *   **Action:** In `crateController.svelte.ts`, define a "no-go" radius around the dragon's center position. Before placing a crate, check if it falls within this radius.
2.  **Adaptive Density & Layout:** Show fewer links on smaller screens to reduce clutter and cognitive load.
    *   **Action:** Use `window.innerWidth` or a media query store to determine the screen category (e.g., 'mobile', 'tablet', 'desktop'). Adjust the `gridWidth` and `maxItems` accordingly.
3.  **Organic Placement (Code Sketch):** For a more dynamic feel, use physics-based or circular packing.
    *   **Action:** Modify `calculateCratePositions` to arrange items in arcs or clusters rather than a rigid grid.
    *   **Code Sketch (`crateController.svelte.ts`):**
        ```typescript
        // A conceptual sketch for arc-based placement
        export function calculateArcPositions(
          count: number,
          category: 'left' | 'right',
          viewport: { width: number; height: number }
        ) {
          const positions = [];
          const radius = viewport.width / 4; // Base radius
          const angleStep = Math.PI / (count * 1.2); // Spacing
          const side = category === 'left' ? -1 : 1;

          for (let i = 0; i < count; i++) {
            const angle = angleStep * i - (Math.PI / 4);
            // Basic check to avoid dragon occlusion zone
            const currentRadius = radius + (i * 0.5); // Stagger radius
            if (currentRadius < DRAGON_NO_GO_RADIUS) continue;

            positions.push({
              x: Math.cos(angle) * currentRadius * side,
              y: 0, // Keep on ground plane
              z: Math.sin(angle) * currentRadius,
            });
          }
          return positions;
        }
        ```

### **5. Metrics and Verification**

*   **How to Measure:**
    *   **Initial Load:**
        *   **Chrome DevTools (Lighthouse):** Run a performance audit to get metrics like LCP, TBT, and a link to the Performance trace.
        *   **Chrome DevTools (Network):** Disable cache, filter by JS, and check the size/load time of your main chunks.
    *   **Runtime Performance:**
        *   **FPS:** Add `@threlte/extras` `<Perf />` component. It provides a simple FPS and memory graph.
        *   **GPU Time:** Use **Spector.js** (browser extension) to capture a single frame and analyze draw calls, shader complexity, and texture memory. This is invaluable for debugging rendering bottlenecks.
*   **A/B Testing with Flags:**
    *   **Mechanism:** Use URL query parameters (`?flag=new_feature`) as your toggle. Create a readable store in Svelte to access these flags throughout the app.
    *   **Process:**
        1.  Deploy a version with the feature flag logic.
        2.  Load the app with the flag enabled (`/page?shadows=contact`).
        3.  Record metrics (FPS, GPU time via Spector.js).
        4.  Load the app without the flag (`/page`).
        5.  Record baseline metrics.
        6.  Compare the results to validate the improvement.
